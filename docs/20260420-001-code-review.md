# Hermes Control Plane Code Review

审查日期：2026-04-20

结论先行：当前仓库**还不是一个可持续演进的双入口控制平面**。`hermes/*` 这条主线已经有六边形架构的雏形，但 `web/backend/*` 又长出了一套平行的 session/core/persistence/runtime。现在的整体形状更像**一把瑞士军刀**，不是工具箱，更不是工业流水线。

一句话判断：

- 如果删掉 `cli/`，`hermes/*` 还能活。
- 如果删掉 `web/backend/*`，主系统也还能活。
- 但如果你想让 CLI 和 Web 真正共享同一控制平面，当前结构**做不到**，因为 Web 入口没有复用主 Core，而是在仓库里另起炉灶。

---

## 1. 入口层与 Core 的物理隔离度

### 总评

CLI 入口勉强接近“薄适配层”，但仍有越界；Web 入口则已经明显越界，且不是适配器，而是**第二套半成品控制平面**。

### CLI 入口

好的部分：

- `cli.py` 本身非常薄，只是转发到 `hermes.interfaces.cli.main()`。
- `hermes/interfaces/cli/adapter.py:370-374` 的启动路径清晰：创建交互端口，调用 `bootstrap()`，运行 CLI。

污染点：

- `hermes/interfaces/cli/adapter.py:110-123` 直接执行 `git rev-parse`。这不是致命问题，但说明 CLI 适配层已经开始自行触达外部进程，而不是纯粹做“解析输入 -> 调 Core -> 渲染输出”。
- `hermes/interfaces/cli/adapter.py:249-282` 直接在入口层编排记忆压缩策略，并用 `asyncio.run(...)` 直接驱动核心异步能力。压缩动作应该是 Core service 的显式用例，不应由 CLI 自己持有流程细节。
- `hermes/interfaces/cli/adapter.py:307-340` 的任务创建逻辑把回调绑成 `lambda: self._console.print(...)`。这意味着任务系统的行为依赖 CLI Console，对 Web/TUI/gRPC 都不可复用。

结论：

- CLI 不是灾难，但已经从 adapter 滑向了“带一点业务编排的终端应用层”。

### Web 入口

这里问题更重，而且是结构性问题。

#### Web Handler 直接操作文件系统

- `web/backend/api/routes/chats.py:16-75` 在路由模块里定义数据目录、创建目录、读写 JSON 文件。
- `web/backend/api/routes/chats.py:101-170` 各个 handler 直接调用这些文件读写函数。

这不是 adapter，这是把 persistence 塞进路由层。按照你的标准，这已经属于入口污染。

#### Web 入口自带状态管理

- `web/backend/main.py:9-13` 在 FastAPI lifespan 里创建 `SessionManager`。
- `web/backend/core/session.py:60-92` 维护 `self.sessions: Dict[str, ChatSession]`。

活跃会话状态没有落在入口无关的 `StateManager` / 应用服务中，而是挂在 Web 进程状态里。CLI 和 Web 根本不共享运行态。

#### Web 入口没有调用主控制平面

- `web/backend/main.py:5-6` 导入的是 `api.routes` 和 `core.session`。
- 它**没有**调用 `hermes.app.bootstrap.bootstrap()`，也没有复用 `hermes.core.agent.AgentSession`、`hermes.services.*`、`hermes.app.settings.Settings`。

这说明 Web 入口不是 `hermes/*` 的 adapter，而是另一套系统的壳。

#### Web 路由甚至存在第二份 WebSocket runtime

- `web/backend/main.py:36-45` 定义了一套 WebSocket 聊天入口，使用 `web/backend/core/session.py`。
- `web/backend/api/routes/ws.py:17-195` 又定义了另一套 `SessionManager`、`active_sessions` 和 WebSocket endpoint。

同一个 Web 应用里出现两套 WebSocket 会话管理，这不是扩展性问题，这是**控制平面分叉**。

#### Web Core 是模拟实现，不是共享 Core

- `web/backend/core/agent_engine.py:62-128` 的 `SimpleAgentRunner` 是一个 demo runner。
- `web/backend/core/session.py:30-57` 和 `web/backend/api/routes/ws.py:72-78` 都在消费这套 Web 专用 runner。

这意味着 Web 不但没有复用主 Core，甚至还复制了一个名字像 Core 的目录来承载临时代码。这个目录名会误导所有后续开发者。

### 可替换性测试

问题答案是：**不能通过**。

如果现在删掉 `cli/` 和 `web/backend/`，`hermes/*` 仍能保留一部分能力；但如果你想把 Web 换成 TUI/gRPC：

- Web 当前依赖 `web/backend/core/session.py`
- Web 当前依赖 `web/backend/core/agent_engine.py`
- Web 当前依赖 `web/backend/api/routes/chats.py` 中的本地 JSON 持久化

这些都不是入口无关的核心模块，因此新入口无法直接复用。

判定：**致命级不通过**。

## 2. 运行时生命周期的管理策略

### `bootstrap()` 偷偷承担了运行时职责

- `hermes/app/bootstrap.py:154-224` 看起来叫 bootstrap，但它不仅构造对象，还：
- `hermes/app/bootstrap.py:201-202` 直接启动 `APSchedulerDriver`
- `hermes/app/bootstrap.py:204-205` 直接加载持久任务

这意味着依赖构建和运行时启动耦在一起。任何入口只要调用 `bootstrap()`，就会隐式启动后台调度器。

问题在于：

- 没有对应的统一 `shutdown()` 对称关闭路径
- 没有 Runtime/Orchestrator 作为显式生命周期边界
- 没有让入口决定“是否启动后台任务系统”

这不是 Unix 风格的清晰入口，而是构造函数式副作用。

### 优雅关闭缺失

- 主系统里能看到 `APSchedulerDriver.shutdown()` 定义在 `hermes/infra/scheduler/apscheduler_driver.py:15-19`
- 但没有看到任何主入口在退出时调用它
- `hermes/interfaces/cli/adapter.py:342-374` 的 CLI 退出路径也没有做资源回收

这意味着调度器生命周期没有被统一托管。

### Web 生命周期也不统一

Web 现在有两套不同的生命周期模型：

- `web/backend/main.py:9-13` 用 FastAPI lifespan 管 `core/session.py`
- `web/backend/api/routes/ws.py:99-195` 自己在 endpoint 内创建/清理 `active_sessions`

这两者之间没有统一调度器、没有统一 agent supervisor、没有统一关闭协议。

### Web 请求与后台任务解耦不足

- `web/backend/api/routes/ws.py:40-44` 用 `asyncio.create_task(self.send_json(data))` 异步发消息

这个 `create_task` 没有追踪、没有取消、没有 drain。连接关闭时，后台发送任务可能还在飞。它未必马上炸，但这是典型的 goroutine/task 泄漏气味。

### 额外的状态模型不稳

- `hermes/services/task_service.py:102` 动态写入 `scheduled.info.last_run`
- 但 `TaskInfo` 在 `hermes/app/ports.py:44-51` 并没有 `last_run` 字段

运行时状态靠“给 dataclass 临时长属性”表达，说明生命周期和状态建模都还没收口。

判定：**不通过**。当前没有统一 Runtime，也没有控制平面级别的生命周期边界。

## 3. 插件机制的解耦程度

### 主线方向是对的

好的部分：

- `hermes/app/ports.py` 定义了 `SkillRepository`、`ToolCatalog`、`SchedulerDriver` 等端口。
- `hermes/infra/persistence/file_skill_repo.py` 负责加载技能文件。
- `hermes/infra/langchain/tools.py` 通过 `SkillRepository` 和 `SkillToolPermissionChecker` 决定工具行为。

这说明主线在试图做到“Core 定义抽象，外部提供实现”。

### 但控制平面对插件的抽象还不够完整

问题：

- 新增 Skill 虽然不一定要改 `core/`，但 `LangChainToolCatalog` 里工具集合是硬编码的：`hermes/infra/langchain/tools.py:172`。
- Core 并没有一个真正的 `Plugin` / `Channel` / `Gateway` 抽象。现在只有 skill metadata，没有运行态插件协议。
- 渠道/会话/Agent 运行时没有统一消息总线。Web 端状态直接放在 dict 里：
  - `web/backend/core/session.py:62`
  - `web/backend/api/routes/ws.py:14`

这不是通过明确消息通道通信，而是通过进程内共享字典保存会话对象。

### 最危险的是依赖方向已经被 Web 分叉破坏

虽然主线没有 `import hermes.plugins.slack` 这种硬耦合，但 `web/backend/*` 自己定义了：

- `web/backend/core/agent_engine.py`
- `web/backend/core/skill_permissions.py`
- `web/backend/core/session.py`

也就是说，插件/渠道相关概念没有被提炼成主控制平面的稳定边界，导致 Web 开始自行复制概念。

判定：**部分通过，但离“可持续演进的插件控制平面”还差一层正式的 runtime/plugin contract。**

## 4. 配置与状态的单一事实来源

### 配置单一事实来源：只完成了一半

好的部分：

- `hermes/app/settings.py:26-54` 至少把环境变量读取集中到了 `Settings.from_env_and_args()`。

问题更大：

- `cli_args` 参数存在，但完全没有被使用：`hermes/app/settings.py:27-30`
- 没有配置文件层
- 没有明确优先级链
- 没有 HTTP request override 如何进入同一配置归一化路径

所以当前并不存在你要求的：

`默认值 < 配置文件 < 环境变量 < CLI Flag / HTTP Request Override`

只有“环境变量 + 默认值”。

### 多入口完全不一致

这是当前最明显的失败点之一。

- CLI 用 `Settings.model_name`：`hermes/app/settings.py:42`
- Web 会话默认模型硬编码成 `"doubao-seed-2.0"`：`web/backend/core/session.py:11`
- Web API 的模型列表也是硬编码数组：`web/backend/api/routes/models.py:16-40`
- WebSocket 消息里可以传 `model`，但它只流入 Web 自己的 `AgentEngine`：`web/backend/api/routes/ws.py:143-156`

也就是说：

- CLI 的 `--model` 机制实际上还没真正存在
- Web 的 `model` 也没有走主配置系统
- 两个入口根本不在同一处代码做处理和校验

这直接不符合“单一事实来源”。

### 状态事实来源同样分裂

- CLI/主线状态在 `ControlPlaneApp` / `AgentSession`
- Web 状态在 `web/backend/core/session.py`
- 另一份 WebSocket 状态又在 `web/backend/api/routes/ws.py:14`
- Chat 持久化状态直接散落在 `web/backend/api/routes/chats.py`

现在没有统一的 `StateManager`。状态是：

- 一部分在内存
- 一部分在路由模块级全局变量
- 一部分在 Web 数据目录
- 一部分在主系统 `.hermes/`

这是典型的多真相源。

判定：**不通过**。

## 5. 可测试性的结构支撑

### 主线是可测的，但还没被真正组织成可测系统

好的部分：

- `AgentSession` 依赖 `AgentBackend` 协议，手动注入，天然可 mock：`hermes/core/agent.py:6-18`
- `TaskService` 依赖 `TaskStore` 和 `SchedulerDriver`，可替换 fake：`hermes/services/task_service.py:15-19`
- `SkillService` 很薄，也容易测

这说明主线有单元测试友好的基础。

### 但测试支撑还没真正落地

- 仓库里没有 `tests/` 目录
- `bootstrap()` 强绑定真实基础设施：
  - `LangChainOpenAIBackend`
  - `LangChainToolCatalog`
  - `APSchedulerDriver`
  - `MemoryManager`

所以如果开发者从应用入口测起，就会被真实模型、真实调度器、真实文件系统、副作用启动拖住。

### Web 端几乎不可复用地测试

- 路由直接读写文件：`web/backend/api/routes/chats.py:22-75`
- WebSocket 逻辑直接维护全局 `active_sessions`：`web/backend/api/routes/ws.py:14`
- `main.py` 直接实例化 `SessionManager`：`web/backend/main.py:9-13`

这些都意味着：

- 你没法只测“控制平面用例”
- 你会被 FastAPI、WebSocket、文件系统状态一起绑住

### DI 方式评价

主线是手动依赖注入，方向正确。

但问题在于应用层没有把“装配”和“运行”继续拆开，导致 `bootstrap()` 成了半个 service locator + 半个 runtime launcher。

判定：**主线有可测潜力，整体系统不可测性仍然偏高。**

## 6. SDK 提取测试：如果要抽出 Library/SDK，我会删什么

### 可以保留，作为 SDK 主体的部分

- `hermes/app/ports.py`
- `hermes/core/*`
- `hermes/services/*`
- `hermes/infra/*`
- `hermes/app/settings.py`（但要重构）
- `hermes/app/bootstrap.py`（但要拆分）

前提是先把 `bootstrap.py` 拆成两层：

- `assemble_control_plane(...)`：纯对象装配，不做副作用启动
- `Runtime` / `Orchestrator`：显式 `start()` / `stop()`

### 我会直接删掉或移出 SDK 的文件

- `cli.py`
- `hermes/interfaces/cli/*`
  - 原因：这是纯 CLI 适配层，不属于 SDK。
- `web/backend/main.py`
- `web/backend/start.py`
- `web/backend/api/routes/*`
- `web/backend/core/*`
  - 原因：这些不是 SDK，而是当前 Web 专用入口和一套分叉 runtime。
- `web/frontend/*`
  - 原因：纯前端。

### 更关键的一刀

如果目标是“可被 Go/TypeScript 项目引用的核心能力”，那我会把 `web/backend/core/*` 整个判定为**不应进入 SDK 的临时层**。因为它不是复用主 Core，而是在复写核心概念。

## 重构路径

只给你的代码提路径，不参考 OpenClaw。

### 第一刀：先消灭双 Core

目标：

- `web/backend/core/*` 全部退役
- Web 只能调用 `hermes/*` 暴露的应用服务

做法：

1. 在 `hermes/app/` 下新增显式用例服务，例如：
   - `ConversationService`
   - `SessionService`
   - `GatewayService`
   - `ModelCatalogService`
2. Web handler 只做：
   - 解析请求
   - 调用服务
   - 映射响应
3. 所有会话状态迁入入口无关的状态仓库或 runtime manager，不再放在 FastAPI 层 dict 里。

### 第二刀：把 bootstrap 拆成“装配”和“运行”

目标：

- 入口决定何时启动调度器、何时关闭后台任务
- 生命周期只在一个地方管理

建议形状：

- `assemble_control_plane(settings, deps...) -> ControlPlane`
- `ControlPlaneRuntime(control_plane, scheduler, ...)`
- `runtime.start()`
- `runtime.stop()`

这样 CLI、Web、TUI、gRPC 都能共享同一个 runtime contract。

### 第三刀：建立统一的状态模型

把下面这些状态收口：

- active sessions
- chat history metadata
- active agents
- model selection
- task execution state

不要再让它们分别散落在：

- `app.state`
- 全局 dict
- 路由模块级变量
- Web 本地 JSON 文件
- `.hermes/` 的不同角落

至少要有：

- `SessionStore` 端口
- `ConversationStore` 端口
- `AgentRegistry` / `RuntimeState` 模块

### 第四刀：配置归一化

建立一个统一入口：

- `ResolvedAgentConfig`
- `ResolvedRuntimeConfig`

由同一个模块负责合并：

- 默认值
- 配置文件
- 环境变量
- CLI flags
- HTTP request overrides

然后 CLI 和 Web 都把原始输入映射成这套对象，再交给 Core。

### 第五刀：把“模型目录”“项目目录”“聊天存储”从 Web 路由里拔掉

尤其是：

- `web/backend/api/routes/models.py`
- `web/backend/api/routes/projects.py`
- `web/backend/api/routes/chats.py`

这些文件现在不是 handler，它们是杂糅了：

- 伪仓储
- 伪配置
- 伪状态管理
- 伪业务逻辑

必须迁到 `hermes/services` + `hermes/infra`，让 Web 只保留 handler。

---

## 最终判词

当前代码库**不是 Web 路由的大杂烩**，因为 `hermes/*` 主线已经有明确的分层意识；但它也**绝对还不是成熟控制平面**，因为 Web 入口绕开主线，自己复制了一套 session/core/runtime/persistence。

最像的形状：**一把瑞士军刀**。

原因不是“功能多”，而是：

- 主刀片 `hermes/*` 正在成形
- 侧面又弹出一组 `web/backend/*` 自制刀片
- 每个刀片都能做一点事，但它们没有围绕同一个中轴收拢

最核心的一句话：

> 你现在最大的架构风险，不是某个模块写得丑，而是仓库里已经出现了“两套控制平面争夺真相源”的趋势。

如果不尽快把 Web 收编为 adapter，后面再加 Gateway、Channel、Agent Supervisor 时，仓库会非常快地滑向你最讨厌的那种“大泥球”。

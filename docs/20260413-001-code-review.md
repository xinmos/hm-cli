# hm-cli 架构审查

## 一句话结论

当前代码**不像一个可持续演进的控制平面**，更像一组被 `cli.py` 临时编排起来的脚本能力集合。  
如果继续在这个形态上叠加 Gateway、Channel、Agent 管理与 Skill 调度，最终会演变成你最厌恶的那种“大泥球”。

一句话归类：**一把瑞士军刀**。功能开始变多，但刀片都装在同一个手柄里，彼此已经开始碰撞。

---

## 1. CLI 与 Core 的物理隔离度

### 结论

这一项当前是**致命级不合格**。

`cli.py` 不是“薄入口”，而是同时承担了：

1. 终端 UI 渲染
2. 应用装配
3. 运行时生命周期启动
4. Skill 发现与加载
5. System Prompt 拼装
6. 调度器管理
7. Agent 调用编排
8. 安全确认交互

这说明你的“控制平面”还没有 Core，只有一个长得像 CLI 的 App。

### 证据

- `cli.py:26-53`
  `HermesCLI.__init__()` 内直接实例化 `Console`、`HermesAgent`、`TaskScheduler`，读取全局 registry，加载技能，启动 scheduler，注册 confirm callback。
- `cli.py:55-69`
  CLI 直接读取 `hermes/prompts/system.md` 文件并拼接运行时 prompt。
- `cli.py:123-155`
  CLI 直接扫描磁盘目录加载技能。
- `cli.py:300-327`
  CLI 直接管理任务的增删查，而不是调用一个 `TaskService` / `Runtime` / `ControlPlane`。

### 架构问题

- 删掉 `cli.py` 并换成 TUI 或 HTTP API 后，Core **不能保持不变**。
- CLI 不是 Adapter，而是 Orchestrator。
- 现在的依赖方向是：`CLI -> 所有东西`，但正确方向应该是：`CLI -> Application/Core Port -> Infrastructure Adapter`。

### 是否存在架构污染

是，而且很明显。

`hermes/agents/agent.py:17-22` 里 Core 直接 new `ChatOpenAI` 并 `create_agent(...)`。  
这意味着所谓的 “Agent Core” 其实已经绑死在 LangChain + OpenAI 兼容接口上。

这就是**架构污染**。  
你的核心域对象不应该知道具体模型 SDK 是 `ChatOpenAI`，更不应该在构造函数里直接连上它。

### 最短重构路径

先拆成三层，而不是一上来搞复杂框架：

1. `cmd/` 或 `interfaces/cli/`
   只保留参数解析、stdin/stdout 交互、把用户输入转成 Application Command。
2. `app/`
   放 `ControlPlane`, `Runtime`, `TaskService`, `SkillService`, `SessionService`。
3. `infra/`
   放 `LangChainAgentRunner`, `FileSkillRepository`, `JsonTaskStore`, `RichConsoleUI`。

CLI 只能依赖 `app` 暴露的 use case，不得直接碰 `Path.read_text()`、`TaskScheduler()`、`get_registry()`、`set_confirm_callback()`。

## 2. 运行时生命周期的管理策略

### 结论

这一项目前是**脆弱且不可扩展**。

你已经有运行时实体了: Agent、Scheduler、Tool 调用、上下文压缩。  
但代码里没有一个显式的 `Runtime` / `App` 对象去统一托管生命周期。

### 证据

- `cli.py:45-47`
  CLI 构造时直接 `self._scheduler = TaskScheduler()` 并立即 `start()`。
- `cli.py:367-384`
  `run()` 退出时没有显式调用 `self._scheduler.shutdown()`。
- `hermes/scheduler.py:35`
  使用 `BackgroundScheduler`，说明内部已经引入后台线程。
- 全仓库没有看到 `signal`、统一关闭流程、`try/finally` 形式的 runtime teardown。

### 具体风险

1. `BackgroundScheduler` 后台线程由 CLI 顺手启动，却没有被对称关闭。
2. 未来加 Gateway/WebSocket/Channel consumer 后，泄漏风险会立刻放大。
3. 优雅退出逻辑没有单一归口，后续一定会散落到 CLI、Tool、Channel、Agent 各处。
4. 任务执行异常现在只 `print()`，没有结构化事件、没有错误上报、没有重试策略。

### 第一性原理判断

控制平面的第一职责不是“把功能跑起来”，而是“**谁创建，谁托管，谁关闭**”。  
你现在缺的是这个托管层。

### 最短重构路径

新增一个最小 `Runtime`：

- `start()`: 启动 scheduler、gateway、channel worker、session store
- `stop()`: 统一关闭后台资源
- `register_signal_handlers()`: 只在 CLI adapter 做系统信号绑定
- `run_forever()` 或 `serve()`: 把阻塞语义放到 runtime，不放在 CLI while loop 里

规则要很硬：

- 信号处理只允许出现在最外层入口
- 资源关闭只能由 `Runtime.stop()` 统一触发
- 不允许某个模块悄悄自启动后台线程

## 3. 插件机制的解耦程度

### 结论

目前只有“可加载文件”，还没有“可插拔架构”。

Skill 层勉强接近插件，但 Tool、Agent、Runtime、Registry 之间的边界并不稳定。  
新增 Skill 不一定需要改核心代码，但新增一种插件能力时，大概率还是要动核心。

### 证据

- `hermes/skills/registry.py`
  `SkillRegistry` 负责解析和注册，但注册中心是进程内全局单例。
- `hermes/skills/__init__.py:7-13`
  `get_registry()` 使用模块级 `_registry`。
- `hermes/tools/skill_loader.py:18`
  Tool 通过全局 `get_registry()` 反查技能。
- `hermes/tools/confirm.py:4-23`
  危险操作确认通过模块级全局回调传递。
- `hermes/tools/__init__.py:5`
  `TOOLS` 是静态列表，新增工具要改核心代码。

### 问题本质

这不是插件系统，而是“全局对象 + 文件扫描 + 静态装配”。

它有几个结构性缺陷：

1. Core 没有定义稳定的插件接口
2. 插件发现机制和运行时注册机制耦合在一起
3. 插件通过全局单例和全局回调共享状态
4. Tool 集合是编译期常量，不是运行时组合

### 依赖方向是否正确

不完全正确。

正确方式应该是：

- Core 定义 `SkillProvider`、`ChannelDriver`、`ToolProvider`、`AgentBackend`
- Infra/Plugin 去实现这些接口

现在的方向更像：

- Core 直接依赖 LangChain Tool 机制
- Tool 再回头依赖全局 SkillRegistry / 全局 confirm callback

这会导致 Core 逐渐反向依赖具体插件实现细节。

### 最短重构路径

最少先定义四个端口：

```text
SkillRepository
ToolCatalog
AgentBackend
InteractionPort
```

然后遵守两个规则：

1. Core 不 import 具体插件实现
2. 插件之间不允许通过全局变量通信，只能通过显式上下文或消息对象通信

如果以后要加 Channel/Gateway，建议直接走显式消息模型：

- `Command`
- `Event`
- `Task`
- `Envelope`

而不是让插件彼此直接拿对象引用。

## 4. 配置与状态的单一事实来源

### 结论

这一项目前是**隐式、分散、不可测试**。

### 证据

- `hermes/config.py:5`
  import 时直接 `load_dotenv()`。
- `hermes/config.py:7-29`
  所有配置都在类属性初始化时通过 `os.getenv()` 读取。
- `hermes/security.py:8`
  模块导入时把 `WORKDIR = Config.WORKDIR` 固化到模块级变量。
- `hermes/scheduler.py:38`
  默认存储路径再次从 `Config.WORKDIR` 推导。
- `cli.py`
  又在多个位置直接读取 `Config.MODEL_NAME`、`Config.WORKDIR`。

### 架构问题

1. 配置没有显式装配点
2. 配置优先级没有被建模，也没有测试入口
3. import 时副作用导致测试和嵌入式调用不可预测
4. 模块级缓存会让配置变更无法动态生效

### 动态状态现状

运行时状态散落在各处：

- Agent 消息在 `HermesAgent._messages`
- 任务状态在 `TaskScheduler._tasks`
- Skill 状态在全局 `_registry`
- confirm 行为在全局 `_confirm_callback`

这不是单一事实来源，而是**多个局部真相并存**。

### 第一性原理判断

控制平面必须区分两类东西：

1. `Config`: 启动时装配，原则上只读
2. `State`: 运行时变化，必须有归口 owner

你现在把这两类东西都做成了“谁方便谁持有”。

### 最短重构路径

配置：

- 用 `Settings` 数据类承载配置
- 在唯一入口完成解析与合并
- 明确优先级: 默认值 < 配置文件 < 环境变量 < CLI 参数
- 不允许模块 import 时读取环境变量

状态：

- 新建 `RuntimeState` 或 `SessionStore`
- Agent session、task registry、channel subscriptions 都从这里挂出
- UI 只能读状态快照，不能直接改内部对象

顺手指出一个实现问题：项目依赖里已经有 `orjson`，但 `hermes/scheduler.py` 仍然使用标准库 `json`，这说明“技术约束”还没有进入架构层。

## 5. 可测试性的结构支撑

### 结论

当前结构对单元测试非常不友好。

如果一个开发者想测试：

- Agent 调用流程
- Skill 加载
- 定时任务管理
- 上下文压缩
- 危险命令确认

他会频繁撞上全局状态、模块副作用和具体 SDK。

### 证据

- `hermes/agents/agent.py:17-22`
  构造函数里直接创建真实 `ChatOpenAI`
- `hermes/context.py:31-36`
  Compressor 默认也直接创建真实 `ChatOpenAI`
- `hermes/tools/__init__.py:5`
  Tool 列表静态绑定
- `hermes/skills/__init__.py`
  全局单例 registry
- `hermes/tools/confirm.py`
  全局回调
- 仓库里没有 `tests/` 目录

### 这意味着什么

你目前的对象大多不是“可注入依赖的组件”，而是“带副作用的现成实例工厂”。

这会直接导致三类后果：

1. 单元测试不得不 mock 模块级对象
2. 测试彼此污染，因为共享全局状态
3. 很难做无网络、无模型、无磁盘的快速测试

### 最短重构路径

最低成本的改法不是上容器，而是**手动依赖注入**：

- `HermesAgent(agent_backend, tool_catalog, context_manager)`
- `ContextCompressor(llm_client)`
- `TaskScheduler(task_store, clock)`
- `SkillService(skill_repo)`

然后在最外层 `bootstrap.py` 里做一次装配。

只要做到这一点，单元测试就能替换成：

- Fake LLM
- InMemoryTaskStore
- Fake SkillRepository
- Stub InteractionPort

这才是控制平面应该具备的可演进性。

## 6. 如果要打包成 Library/SDK，我会删什么

### 先删掉或下沉的文件

1. `cli.py`
   它是终端交互层，不应进入 SDK。
2. `hermes/tools/confirm.py`
   这是 UI 交互策略，不是核心能力。
3. `rich` / `prompt_toolkit` 相关依赖和表现逻辑
   SDK 不应该假设调用方是终端。
4. `hermes/prompts/system.md`
   如果保留，也应降级为默认资源，由调用方显式注入。

### 必须重写或拆分的文件

1. `hermes/agents/agent.py`
   拆成 `AgentSession` 与 `AgentBackend`。前者是 Core，后者是 LangChain/OpenAI Adapter。
2. `hermes/config.py`
   改成显式 `Settings` 装配模块。
3. `hermes/tools/__init__.py`
   改成可组合的 `ToolCatalog`，不能再是静态数组。
4. `hermes/skills/__init__.py`
   去掉全局 registry。
5. `hermes/scheduler.py`
   拆成 `TaskService` + `TaskStore` + `SchedulerDriver`。

### 可以保留为 Core 雏形的部分

1. `hermes/skills/registry.py`
   可以保留“技能元数据解析”这部分，但不能继续承担全局注册中心职责。
2. `hermes/security.py`
   可以保留命令安全判定规则，但 `WORKDIR` 不应在模块导入时固化。

## 最关键的重构路线

不要大修，按顺序切：

### 第一刀：先建立装配边界

新增 `hermes/app/bootstrap.py`：

- 解析 settings
- 构建 repositories / backends / runtime
- 返回 `ControlPlaneApp`

`cli.py` 只做：

- 读取输入
- 调用 `app.handle(command)`
- 渲染输出

### 第二刀：把模型 SDK 赶出 Core

定义接口：

```python
class AgentBackend(Protocol):
    def stream(self, messages: list[Message]) -> Iterable[AgentEvent]: ...
```

`LangChainOpenAIBackend` 放到 `infra/`。  
`HermesAgent` 只能依赖 `AgentBackend`，不能依赖 `ChatOpenAI`。

### 第三刀：消灭全局单例

依次去掉：

- `Config` 类属性式配置
- `get_registry()`
- `_confirm_callback`
- `TOOLS` 静态全局列表

替换成显式构造与传参。

### 第四刀：建立统一 Runtime

把以下资源统一放入 `Runtime`：

- scheduler
- gateway/channel worker
- agent sessions
- task dispatch loop

任何后台资源都只能由 `Runtime` 创建和销毁。

### 第五刀：把 Skill/Tool/Channel 都变成 Port + Adapter

不要把“技能文件解析”“LangChain Tool”“频道接入”“CLI 命令”混成一类抽象。  
它们是四种不同层级的概念。

---

## 最后判断

如果今天停在这里，这个项目还能作为个人原型继续长；  
但如果目标真的是“AI 智能体 CLI 的控制平面”，现在这套结构**还没进入可持续演进区间**。

最该警惕的不是代码少，而是**边界还没建立就开始叠功能**。  
一旦 Gateway、Channel、多 Agent、插件生命周期、远程状态同步进来，`cli.py` 这种入口型上帝对象会立刻失控。

直接一点说：

- `cli.py` 现在是上帝对象
- `Config` 是隐式全局环境
- `HermesAgent` 是被 LangChain/OpenAI 污染的伪核心
- `SkillRegistry` 和 `confirm` 机制依赖全局单例
- `TaskScheduler` 被启动了，但没有进入统一 runtime 托管

这不是控制平面。  
这是一个已经长出控制平面野心、但骨架仍停留在脚本阶段的原型。

如果只做一件事，请先做这件事：

**把 CLI 降级为 Adapter，把 Runtime 提升为系统真正入口。**
